# zkSPARQL-bench

A benchmark for zero-knowledge SPARQL queries over verifiable credentials.

This repo serves two roles:

1. **Dataset + query corpus** — generators for LUBM, BSBM, SP2B with
   credentials wrapping the dataset (legacy, published as
   `zksparql-bench` on npm).
2. **Axial benchmark harness** (`harness/`) — drives
   `@jeswr/sparql-noir` across a five-axis cross-product to produce
   the §8.3 / §8.4 / §9.1 figures of the paper. See
   [Axial harness](#axial-harness) below.

### Axial harness

The harness sweeps a **five-axis cross-product** and emits a JSON
result row per cell. Most cells are `n/a` today; they fill in as
new circuits / commitment shapes ship in
[`circuits/sparql_noir`](https://github.com/jeswr/sparql_noir).

| Axis | Values | Notes |
|---|---|---|
| Commitment | sorted-leaf Merkle, prefix-tree | prefix-tree pending (paper §4.3 round 4) |
| Signature | ed25519, schnorr, ECDSA-p256, ECDSA-k1, BBS+, SD-JWT-VC, ML-DSA | first four wired; rest pending paper §4.5 / §7.4 / §7.5 |
| Query class | BGP, FILTER, OPTIONAL, aggregate, path, EXISTS, NOT EXISTS, MINUS | BGP + FILTER wired; rest sparql_noir round-3 / round-4 |
| Dataset | LUBM-1, LUBM-10, BSBM-1k, BSBM-10k, BSBM-100k | LUBM-1 seeded locally; rest require `npm run build` (Docker) |
| Metric | prove-time (ms), verify-time (ms), proof-size (B), circuit-gates | gate-count requires UltraHonk introspection — pending |

Cross-product cardinality: `2 × 7 × 8 × 5 × 4 = 2240` cells per query
(times the number of curated queries). The paper §8.3 reports a
**curated subset**; the harness produces all of them and marks
unsupported cells `n/a`.

#### Bootstrap cell

The simplest fully-wired combination is

```
sorted-leaf-merkle  ×  ed25519  ×  bgp/lubm-q1  ×  LUBM-1  ×  prove-time-ms
```

which is what `npm run bench:bootstrap` executes end-to-end.

#### Local: one cell

```sh
npm install
npm run bench:bootstrap          # writes results/local-bootstrap.json
```

This generates one real prove-time number on your machine. Per
[`feedback_offload_to_ci.md`](https://github.com/jeswr/zkp-sparql-workspace),
local runs are for development only — do not gather paper numbers
locally.

#### CI: cross-product enumeration + bootstrap

`.github/workflows/bench.yml` runs on every PR:

1. enumerate the full cross-product (no compute);
2. execute the bootstrap cell;
3. upload the result JSON as a workflow artefact.

#### Cross-product enumeration

```sh
npm run bench:enumerate          # JSON-lines, one cell per line, no compute
```

#### Full sweep (EC2 / heavy CI)

The full sweep currently requires:

- `dist/{lubm,bsbm,sp2b}/data-signed/` (run `npm run build`, needs Docker);
- AWS credentials per
  [`decisions/aws-credentials-setup.md`](https://github.com/jeswr/zkp-sparql-workspace/blob/main/decisions/aws-credentials-setup.md)
  Step 6 (not yet provisioned — heavy sweep deferred to camera-ready).

Until then, omitted dataset rows render as `n/a` in the §8.3 table.

### Reproducibility

Every result row carries a provenance block:

- `sparqlNoirCommit` — git SHA of the `@jeswr/sparql-noir` build under test;
- `benchRepoCommit` — git SHA of `zkSPARQL-bench`;
- `queryVersion` — sha256 prefix of the query file;
- `hardware` — CPU model / arch / RAM / Node version;
- `date` — UTC ISO-8601 timestamp.

The paper §8.3 cites the specific `sparqlNoirCommit` and the
`benchRepoCommit` produced by the EC2 sweep that lands the
camera-ready numbers.

### Paper integration

The §8.3 LaTeX table is generated by

```sh
node analysis/report.js results/<run>.json --out analysis/out/table-8.3.tex
```

and ingested by the paper as

```latex
\input{paper/sources/zkSPARQL-bench/analysis/out/table-8.3.tex}
```

`analysis/plot.js` is a stub for the two §8.3 figures (prove-time
bar chart + gate-count heatmap) — wired once we have ≥10 cells.

## Legacy: dataset / query generator

The benchmark dataset is generated and released to
https://registry.npmjs.org/zksparql-bench/-/zksparql-bench-X.Y.Z.tgz.

- the dataset can be found in the credentials contained in `dist/BENCHMARK_NAME/data-signed/`
- the `documentLoaderContent.json` gives all URL resolutions that need to be overridden when performing the benchmark

To regenerate locally:

```sh
npm i
npm run build
```

Requirements:

* Docker
* Nodejs (>= 22.x)

### SP2B queries

Fetch and generate the SP2B query files into `sp2b/queries`:

```
npm run sp2b:queries
```

They will be copied into `dist/sp2b/queries` during `npm run build`.
